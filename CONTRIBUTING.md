## How Can I Contribute?

### Reporting Issues

Found something wrong? Here's how to report it:

1. **Check existing issues** to avoid duplicates
2. **Use the issue template** if available
3. **Be specific** - include:
   - What you expected to happen
   - What actually happened
   - Steps to reproduce
   - Your system info (OS, device model, etc.)

### Suggesting Enhancements

Have an idea to make this guide better?

1. **Open an issue** with the tag `enhancement`
2. **Describe your suggestion** clearly
3. **Explain why** it would be useful
4. **Provide examples** if applicable

### Improving Documentation

Documentation improvements are always welcome:

- **Fix typos or grammar**
- **Add clarity to confusing sections**
- **Add screenshots or diagrams**
- **Translate to other languages**
- **Add troubleshooting tips**

### Adding Screenshots

We'd love to have visual examples! If you have:

- Photos of your M5Stick running NerdMiner
- Screenshots of configuration screens
- Diagrams explaining concepts

Please contribute them!

**Image Guidelines:**
- Clear, well-lit photos
- Crop to show relevant parts only
- PNG or JPG format
- Maximum 1MB file size
- Add alt text for accessibility

## Pull Request Process

1. **Fork the repository**
2. **Create a branch** from `main`:
````bash
   git checkout -b feature/your-feature-name
````
3. **Make your changes**
4. **Test thoroughly** - make sure instructions work
5. **Commit with clear messages**:
````bash
   git commit -m "Add troubleshooting section for WiFi issues"
````
6. **Push to your fork**:
````bash
   git push origin feature/your-feature-name
````
7. **Open a Pull Request** against the `main` branch

### Pull Request Guidelines

- **One feature per PR** - keep changes focused
- **Update README** if adding new sections
- **Follow existing formatting** - keep it consistent
- **Test your instructions** before submitting
- **Write clear commit messages**

## Style Guide

### Markdown Formatting

- Use ATX-style headers (`#` not underlines)
- Use fenced code blocks with language specified:
````markdown
```bash
  your command here
```
````
- Use **bold** for emphasis, *italic* for technical terms
- Add emojis to headers for visual appeal (but don't overdo it)

### Code Style

- Use proper indentation (2 spaces for most files)
- Include comments for complex operations
- Test all commands before including them

### Writing Style

- **Be beginner-friendly** - assume minimal technical knowledge
- **Use clear, simple language**
- **Break complex steps** into smaller chunks
- **Add examples** whenever possible
- **Explain why**, not just how
